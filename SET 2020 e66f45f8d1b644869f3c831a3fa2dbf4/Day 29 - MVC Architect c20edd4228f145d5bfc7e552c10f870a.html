MVC - Model-View-Controller

View ⇒ HTML, JS

  • VD: Convert HTML ⇒ PDF xong trả về frontend (làm trên server)
  • SEO: Search Engine Optimization ⇒ dùng Server Side Rendering (SSR) : khi gọi file html thì đã có nội dung trong file sẵn rồi. VD: Settings, account, etc của FB
  • Hiện tại đang dùng Client Side Rendering (CSR) : khi mở http file lên xong file http gọi api rồi mới load nội dung lên http. VD: Newsfeed FB
  • Việc quyết định khi nào dùng cái gì sẽ do Architect làm
  • Sự khác biệt giữa server-side rendering và client-side rendering

Router(bộ định tuyến)

Xử lý request, URL,... để trỏ đến đúng Controllervà middlewares(nếu có). Nếu không có controller khả dụng thì thông báo không có. Cách dùng: Khai báo routes là mảng gồm pathname, method,.. tên controller rồi viết hàm xử lý request, URL gọi controller/middlewares hoặc dùng switch case if else

  • VD: Kiếm ko đc router ⇒ Status code 404
  • VD: ko vào tới đc server ⇒ Status code 502

Interceptor

Những hàm được chạy trước Controller, thường là: xử lý validate, authenticate, authorize hoặc ghi log trc khi request được truyền tới Controller

var a = "function x() {console.log('abc')};x()" eval(a)

⇒ nguy hiểm

Controller

Xử lý chính, nhận request được điều hướng từ router rồi gọi Repository xử lý xong thì response

Cách dùng: index chứa các hàm(API) của project. Xử lý request và gọi các hàm con ở helper.js để giao tiếp với Repository thực hiện chức năng rồi trả kết quả về response

Repository

Nhận các yêu cầu từ Controllerrồi gọi DataSource để đọc/ghi và xử lý dữ liệu

Không thực hiện trực tiếp đọc ghi database mà chỉ gọi DataSource làm. Khi cần đổi database thì chỉ cần viết lại DataSource bằng cú pháp tương ứng với database mới, giữ nguyên tên hàm nên không phải sửa lại cả Project

1 object nhận giá trịtên Table của CSDL và Model và có các phương thức chính giao tiếp với DataSourceđể xử lý

createuser, deleteuser, filteruser, update

  • POST /sign-up →router→controller(index→helper)-> Repository (validate(data,model) -> createUser(data)

DataSource

Chứa các phương thức truy cập Database chính như (Đọc, Ghi) phù hợp với databaseđược dùng Khi cần đổi database thì chỉ cần thay đổi cú pháp đọc ghi tương ứng với database mới

  • để xác định được đang sử dụng database nào
  • Multitenant: nhiều cty và mỗi cty có 1 database riêng nhưng có chung 1 cấu trúc (VD: discord, slack, workplace, social network) ⇒ khó code nhưng nếu cần thì vẫn phải làm
  • Dùng lúc nào sẽ do Architect quyết định
  • Microservice: chia nhỏ app thành nhiều khối, mỗi khối làm 1 tác vụ riêng và là 1 cái nodejs riêng (máy riêng)
  • Ngoài lề: IoT hay dùng DynamoDB ⇒ gửi nhận liên tục ko vấn đề gì
